El club tiene 78 amistades reales. Pero en el grafo guardamos cada amistad en ambas direcciones:

📦 ¿Por qué se llama "COO Format"?

COO = Coordinate Format (Formato de Coordenadas)

Es como dar direcciones GPS para cada conexión. En lugar de guardar una matriz gigante de 34×34 con muchos ceros, solo guardamos las coordenadas donde hay conexiones:

❌ Sin COO (matriz densa)
34 × 34 = 1,156 valores
¡Muchos ceros innecesarios!
✅ Con COO (edge_index)
2 × 156 = 312 valores
¡Solo guardamos las conexiones!

🚨 ¡CUIDADO! Confusión muy común: Matriz de Features (X) vs Matriz de Adyacencia (A)

Ambas son matrices de 34 × 34 en este dataset. Ambas tienen 1s y 0s. Pero representan cosas completamente diferentes. Es como confundir un pasaporte con una lista de contactos: ambos son documentos con nombres, ¡pero uno dice quién eres y el otro a quién conoces!

⚖️ Resumen: Las 5 diferencias clave

Aspecto 📋 Matriz de Features (X) 🔗 Matriz de Adyacencia (A)
¿Qué guarda? Las propiedades de cada nodo
(quién es)		 Las conexiones entre nodos
(a quién conoce)
Filas son... Nodos Nodos
Columnas son... Características
(edad, intereses, etc.)		 Otros nodos
(posibles amigos)
X[i][j] = 1 significa... "El nodo i tiene la característica j" "El nodo i es amigo del nodo j"
Patrón de 1s Diagonal ↘️
(en Karate Club)		 Dispersa y simétrica 🔀
(donde hay amistades)
La diagonal A[i][i]... Siempre = 1
(me identifico a mí mismo)		 Siempre = 0

Imagina una red social con 3 usuarios: Ana (nodo 0), Bob (nodo 1), Carlos (nodo 2).

"¿Qué atributos tiene cada persona?"

Ana Bob Carlos

🎯 Regla de oro:

📋 X responde → "Ana es gamer y chef" (características propias)
🔗 A responde → "Ana es amiga de Bob" (relaciones con otros)

¡La GNN necesita AMBAS! Usa X para saber qué información tiene cada nodo, y A para saber a quién preguntarle información.

⚠️ ¿Por qué se confunden en el Karate Club?

¡Porque AMBAS son de 34×34 y ambas tienen 1s y 0s! Pero es pura coincidencia:

  • X es 34×34 porque hay 34 nodos y cada uno usa one-hot encoding de tamaño 34 (no hay features reales, solo un ID)
  • A es 34×34 porque hay 34 nodos y cada celda dice si dos nodos están conectados

En un dataset real, tendrían tamaños diferentes:
Ejemplo: 1000 usuarios con 50 features → X = [1000, 50] pero A = [1000, 1000]

💡 Truco para no confundirte: Las columnas de X son características (edad, género...). Las columnas de A son otros nodos. ¡Son conceptos totalmente diferentes!

🤔 ¿Qué diablos es un "mask" y para qué sirve? — Explicación profunda

Imagina que eres profesor y tienes una clase de 34 alumnos. Quieres hacer un examen, pero SOLO conoces las respuestas correctas de 4 alumnos (los que vinieron a tutoría). Tu reto es: usando lo que sabes de esos 4, ¿puedes predecir las notas de los otros 30?

📋 ¿Qué es técnicamente un mask?

Un mask (máscara) es simplemente un tensor de valores booleanos (True/False) con la misma longitud que el número de nodos. Funciona como un filtro:

Los 34 nodos del grafo:
Nodo: 0 1 2 3 4 5 ... 24 ... 33
train_mask: ✅ ❌ ❌ ❌ ✅ ❌ ... ✅ ... ✅
Grupo (y): 1 1 1 1 3 3 ... 2 ... 0

Los 4 nodos con True son: nodo 0 (grupo 1), nodo 4 (grupo 3), nodo 24 (grupo 2), y nodo 33 (grupo 0).
¡Exactamente un representante por cada clase!

🔧 ¿Cómo se usa en el código? — Indexación booleana

El mask se usa como índice de un tensor. Esto se llama indexación booleana (boolean indexing) y es una técnica fundamental de PyTorch/NumPy:

# Sin máscara: usamos TODOS los nodos
loss = criterion(z, data.y)
→ z tiene forma [34, 4] (34 nodos, 4 clases)
→ data.y tiene forma [34] (etiqueta de cada nodo)

# Con máscara: usamos SOLO los 4 nodos de entrenamiento
loss = criterion(z[data.train_mask], data.y[data.train_mask])
→ z[train_mask] tiene forma [4, 4] (solo los 4 nodos True)
→ data.y[train_mask] tiene forma [4] (solo sus etiquetas)
❌ Sin mask (train en TODOS)
z = predicciones de los 34 nodos
y = etiquetas de los 34 nodos
loss = error sobre 34 nodos

⚠️ El modelo "hace trampa" porque ve las respuestas de TODOS durante el entrenamiento
✅ Con mask (train solo en 4)
z[mask] = predicciones de solo 4 nodos
y[mask] = etiquetas de solo 4 nodos
loss = error sobre solo 4 nodos

✅ Los otros 30 nodos son el "examen sorpresa"

🎭 Analogía: El examen parcial vs. el examen final

🏫 Imagina una escuela con 34 estudiantes divididos en 4 casas (como Hogwarts):

📝 train_mask = True → Son los 4 estudiantes que llevan el uniforme de su casa. Tú SABES a qué casa pertenecen.

🤷 train_mask = False → Son los 30 estudiantes que vienen de civil. Tú NO sabes su casa, pero puedes adivinarla observando con quién se juntan (las conexiones del grafo).

🧠 El modelo aprende: "Si el estudiante 0 (casa Azul ✅) se junta mucho con el estudiante 1, el estudiante 2 y el estudiante 3... entonces probablemente ellos también son de la casa Azul." ¡Eso es la GNN usando la estructura del grafo!

🔍 ¿Por qué NO se usa train_mask en el notebook del Capítulo 1?

Si miras el código del training loop en este notebook, verás que dice:

loss = criterion(z, data.y) ← ¡Usa TODOS los nodos!

Esto es intencional para simplificar la primera lección. En el Capítulo 2 (GAT) y Capítulo 3 (GraphSAGE) verás el uso correcto:

loss = criterion(out[data.train_mask], data.y[data.train_mask])

En producción SIEMPRE se usan las máscaras para separar train / validation / test correctamente.

🎯 Los 3 tipos de mask en un grafo

En datasets reales, PyTorch Geometric incluye 3 máscaras para hacer un split correcto:

🏋️
train_mask
Nodos para aprender
El modelo ve sus etiquetas
y ajusta sus pesos
🔧
val_mask
Nodos para ajustar
Elegir hiperparámetros
(learning rate, capas, etc.)
🧪
test_mask
Nodos para evaluar
NUNCA se ven durante
el entrenamiento

⚡ ¿Por qué es DIFERENTE a train/test split en ML clásico?

En Machine Learning clásico (tabular), separas los DATOS en conjuntos independientes: el modelo de entrenamiento no sabe que existen los datos de test.

En grafos es diferente: ¡el modelo ve TODOS los nodos durante el forward pass! Todos los nodos participan en el message passing (propagación de información por las aristas). Lo que cambia es:

📊 La función de pérdida (loss) solo se calcula sobre los nodos de train_mask
📊 Los gradientes solo se calculan basándose en esos nodos
📊 Pero la información fluye por TODO el grafo — los nodos de test ayudan a propagar información aunque no contribuyen al loss

Esto se llama "transductive learning" — el modelo tiene acceso a la estructura completa del grafo, pero solo entrena con las etiquetas de unos pocos nodos.

⚠️ ¡Dato clave para recordar!

Solo 4 de 34 nodos tienen True — eso es apenas el 11.8% de los datos.
Hay exactamente un nodo por cada clase (grupos 0, 1, 2, 3).

Esto significa que el modelo debe aprender de solo 4 ejemplos y predecir correctamente los otros 30.

🤯 ¡Es por eso que las GNNs son tan poderosas! Un modelo de ML tradicional con solo 4 ejemplos sería inútil. Pero las GNNs usan la estructura de conexiones del grafo (quién es amigo de quién) para propagar información y generalizar. Es como preguntar: "Si todos tus amigos son del grupo Azul, ¿de qué grupo eres probablemente tú?"

Arquitectura del Modelo GCN

¡Hora de construir nuestra primera red neuronal para grafos! 🏗️

▶️

¡Ejecuta la Celda 11! Aquí definimos la arquitectura de nuestra GNN. Son solo 2 capas, pero hacen magia. 🪄

from torch.nn import Linear
from torch_geometric.nn import GCNConv

class GCN(torch.nn.Module):
    def __init__(self):
        super().__init__()
        self.gcn = GCNConv(dataset.num_features, 3)  # 34 → 3
        self.out = Linear(3, dataset.num_classes)   # 3 → 4

    def forward(self, x, edge_index):
        h = self.gcn(x, edge_index).relu()
        z = self.out(h)
        return h, z

model = GCN()
print(model)

)

📊 El Flujo de Datos: De 34 números a 1 etiqueta

🔮 GCNConv: La Capa Mágica (explicada con ejemplos)

🤔 ¿Qué hace GCNConv que una capa normal no puede hacer?

Una capa normal (Linear) solo mira los datos de UN nodo a la vez.

GCNConv mira los datos del nodo + los datos de TODOS sus vecinos, y los combina.

❌ Capa Linear Normal

"Miro a la persona 0...
Solo sé que es la persona 0.
No sé nada de sus amigos.
¿Cómo adivino su grupo? 🤷"

✅ Capa GCNConv

"Miro a la persona 0...
También miro a sus amigos 1, 2, 3, 4.
La mayoría son del grupo azul.
¡Persona 0 es grupo azul! 🎯"

❓ Preguntas que probablemente tengas

🔢 ¿Por qué 34 → 3 → 4?

34: Cada nodo empieza con 34 números (su "DNI" único).
3: Los comprimimos a 3 para poder visualizarlos en 3D (¡verás esto más adelante!).
4: Hay 4 grupos posibles en el club.

⚡ ¿Por qué ReLU? ¿No podríamos saltarlo?

Sin ReLU, la red solo podría aprender patrones lineales (líneas rectas).
ReLU añade "curvas" al modelo, permitiendo aprender patrones más complejos.
Analogía: Es como la diferencia entre dibujar solo con regla vs. dibujar a mano alzada.

🎯 ¿Por qué el score más alto gana?

Los scores son como "votos de confianza". Si la clase 0 tiene score 2.3 y las demás tienen menos,
el modelo está diciendo: "Estoy más seguro de que es clase 0".
La función argmax simplemente elige el índice con el valor más alto.

🔄 ¿La red "ve" el grafo completo o solo los vecinos directos?

Con 1 capa GCNConv: Cada nodo ve a sus vecinos directos (a 1 salto).
Con 2 capas GCNConv: Vería a vecinos de vecinos (a 2 saltos).
Nosotros usamos 1 capa porque el grafo es pequeño y los grupos están bien conectados.

🌟 CAPA 2: GCNConv — El Corazón de la GCN

Esta es la capa que hace "mágica" a nuestra red. Sin ella tendríamos una red neuronal normal que mira a cada persona del club de forma aislada. GCNConv cambia todo: antes de decidir nada, le pregunta a los amigos de cada persona "¿tú cómo eres?" y usa esa información.

Imagina que llegas de nuevo a una escuela y no conoces a nadie. Quieres saber a qué "grupo" pertenece cada estudiante (deportistas, artistas, gamers, académicos...).

❌ Sin GCNConv:

Miras a Juan. Solo ves su ropa.
Miras a María. Solo ves su ropa.
Decides en aislamiento, sin saber quién es amigo de quién.

✅ Con GCNConv:

Miras a Juan. Ves su ropa + la de sus 5 amigos.
Todos llevan camiseta de fútbol → ¡Juan es deportista!
Usaste el contexto social para decidir.

💡 GCNConv = mirar a tus amigos para entenderte mejor. Eso es literalmente todo. Ahora veamos cómo lo hace matemáticamente.

📐 La Fórmula Matemática — Paso a Paso (con LaTeX)

La fórmula completa de GCNConv según el paper original de Kipf & Welling (2017):

$$\mathbf{h}_i^{\prime} = \sigma \!\left( \sum_{j \in \mathcal{N}(i)\,\cup\,\{i\}} \frac{1}{\sqrt{\hat{d}_i \cdot \hat{d}_j}} \; \mathbf{W} \, \mathbf{h}_j \right)$$

No te asustes — vamos a desmenuzar cada símbolo uno por uno 👇

🔤 Diccionario de Símbolos:
$\mathbf{h}_i^{\prime}$
El RESULTADO — el nuevo vector del nodo $i$

Antes de la capa, el nodo $i$ tenía un vector de 34 números (su identidad). Después de la capa, tendrá un vector de 3 números que incorpora información de sus amigos. El apóstrofe (prima) $'$ significa "la versión nueva".

$\sigma(\cdot)$
Función de activación (ReLU en nuestro caso)

Es la "no-linealidad" que se aplica después de todo lo demás. En nuestro código es .relu(). Sin ella, apilar muchas capas sería inútil — varias transformaciones lineales equivalen a una sola. $\sigma$ añade "curvas" al aprendizaje.

$\displaystyle\sum_{j \in \mathcal{N}(i) \cup \{i\}}$
"Suma sobre todos los vecinos de $i$, incluyéndose a sí mismo"

$\mathcal{N}(i)$ = el vecindario del nodo $i$ (sus amigos en el grafo).
$\cup \{i\}$ = "y también el propio nodo $i$" (esto se llama self-loop).
$j$ es una variable que recorre cada vecino, uno por uno.

🥋 En el Karate Club: El Nodo 0 (Mr. Hi) tiene 16 amigos. Entonces la suma recorre 17 nodos: los 16 amigos + él mismo.

📌 Antes de seguir... ¿Qué es el "grado" de un nodo?

El grado de un nodo es simplemente cuántas conexiones (aristas) tiene. Dicho de forma más sencilla: cuántos amigos tiene esa persona.

🔢 Definición formal:

$$\text{grado}(i) = d_i = \text{número de aristas conectadas al nodo } i$$ Es decir: contar cuántas líneas salen de ese nodo en el grafo.

🥋 En el Karate Club:

• Mr. Hi (Nodo 0): grado = 16 (conoce a 16 personas)
• Nodo 2: grado = 10 (conoce a 10 personas)
• Nodo 11: grado = 1 (solo conoce a 1 persona)
• Nodo 33 (Officer): grado = 12 (conoce a 12)

🎯 Analogía ultra-simple:

Abre tus contactos de WhatsApp. Cuenta cuántos contactos tienes. Ese número = tu "grado" en la red social. Si tienes 200 contactos, tu grado es 200. Si solo tienes 5, tu grado es 5. El grado es solo eso: cuántas conexiones tiene alguien.

💡 ¿Y $\hat{d}_i$ (d-sombrero)? Es el grado + 1: $\hat{d}_i = d_i + 1$. El "+1" es porque en GCNConv cada nodo se conecta consigo mismo (self-loop), así que técnicamente tiene una conexión extra. Si Mr. Hi tiene 16 amigos, $\hat{d}_0 = 16 + 1 = 17$.

$\dfrac{1}{\sqrt{\hat{d}_i \cdot \hat{d}_j}}$
Factor de normalización — el "ecualizador de volumen"

Ahora que sabes qué es el grado, este factor es fácil de entender:

$\hat{d}_i$ = grado del nodo $i$ + 1  (ejemplo: Mr. Hi → $16 + 1 = 17$)
$\hat{d}_j$ = grado del vecino $j$ + 1  (ejemplo: Nodo 1 → $9 + 1 = 10$)

Entonces: $\dfrac{1}{\sqrt{17 \cdot 10}} = \dfrac{1}{\sqrt{170}} \approx 0.077$

¿Para qué sirve? Imagina una fiesta: una persona que conoce a 50 personas (grado 50) envía un mensaje "diluido" — su atención se reparte entre muchos. Una persona que solo conoce a 3 (grado 3) envía un mensaje más "concentrado" y personal. Sin esta normalización, los nodos populares (como Mr. Hi con grado 16) dominarían todo con señales gigantes y los nodos tímidos (como el Nodo 11 con grado 1) serían ignorados.

$\mathbf{W}$
Matriz de pesos — LO QUE LA RED APRENDE

Es una tabla de números de tamaño $34 \times 3$ (en nuestro caso). Inicialmente son valores aleatorios, pero durante el entrenamiento la red los ajusta para que la transformación $\mathbf{W} \cdot \mathbf{h}_j$ produzca representaciones útiles.

Analogía: $\mathbf{W}$ es como un "traductor". Toma el vector largo de 34 números y lo comprime a 3 números que capturan lo esencial.

$\mathbf{h}_j$
El vector actual del vecino $j$

Son los datos que el vecino $j$ tiene antes de esta capa. En nuestro caso es una fila de la matriz identidad (34 números: un "1" y treinta y tres "0s").

🥋 Ejemplo: $\mathbf{h}_3 = [0, 0, 0, 1, 0, \ldots, 0]$ — el Nodo 3 tiene un "1" en la posición 3 y ceros en todas las demás.

📖 Leyendo la Fórmula de Adentro hacia Afuera:
1
Primero: Toma el vector del vecino $j$ → $\mathbf{h}_j$ (34 números)
"¿Cómo es esta persona?"
2
Segundo: Multiplica por los pesos → $\mathbf{W} \cdot \mathbf{h}_j$ (34 → 3 números)
"Traduce su información a un resumen de 3 dimensiones."
3
Tercero: Escala por importancia → $\dfrac{1}{\sqrt{\hat{d}_i \cdot \hat{d}_j}} \cdot \mathbf{W} \, \mathbf{h}_j$
"Ajusta el volumen: si tiene muchos amigos, baja el volumen de su mensaje."
4
Cuarto: Suma los mensajes de TODOS los vecinos → $\displaystyle\sum_{j}$
"Junta toda la información del vecindario en un solo vector."
5
Quinto: Aplica la activación → $\sigma(\ldots) = \text{ReLU}(\ldots)$
"Quita los valores negativos para añadir no-linealidad."
🔢 Ejemplo Numérico Real: Nodo 0 (Mr. Hi) en el Karate Club

Vamos a calcular $\mathbf{h}_0'$ a mano para que la fórmula deje de ser abstracta. Usaremos solo 3 vecinos para simplificar (en realidad Mr. Hi tiene 16).

# Datos del Karate Club (simplificado)
Nodo 0: grado = 16 → d̂₀ = 16 + 1 = 17
Nodo 1: grado = 9  → d̂₁ = 9 + 1  = 10
Nodo 2: grado = 10 → d̂₂ = 10 + 1 = 11
Nodo 3: grado = 6  → d̂₃ = 6 + 1  = 7
📏 Paso 1 — Calcular los coeficientes de normalización

Para no escribir $\frac{1}{\sqrt{\hat{d}_i \cdot \hat{d}_j}}$ cada vez, vamos a darle un nombre corto. Definimos:

📌 Definimos $c_{i,j}$ como atajo del factor de normalización de la fórmula:

$$\boxed{c_{i,j} \;\overset{\text{def}}{=}\; \frac{1}{\sqrt{\hat{d}_i \cdot \hat{d}_j}}}$$

Es solo un nombre, no un símbolo nuevo. Donde en la fórmula ves $\frac{1}{\sqrt{\hat{d}_i \cdot \hat{d}_j}}$, aquí diremos $c_{i,j}$ para que sea más fácil de leer.

Ahora sí, calculemos $c_{i,j}$ para cada par (Nodo 0, vecino $j$):

$$c_{0,0} = \frac{1}{\sqrt{\hat{d}_0 \cdot \hat{d}_0}} = \frac{1}{\sqrt{17 \cdot 17}} = \frac{1}{17} \approx 0.059 \quad \text{(self-loop: } j = i \text{)}$$ $$c_{0,1} = \frac{1}{\sqrt{\hat{d}_0 \cdot \hat{d}_1}} = \frac{1}{\sqrt{17 \cdot 10}} = \frac{1}{\sqrt{170}} \approx 0.077$$ $$c_{0,2} = \frac{1}{\sqrt{\hat{d}_0 \cdot \hat{d}_2}} = \frac{1}{\sqrt{17 \cdot 11}} = \frac{1}{\sqrt{187}} \approx 0.073$$ $$c_{0,3} = \frac{1}{\sqrt{\hat{d}_0 \cdot \hat{d}_3}} = \frac{1}{\sqrt{17 \cdot 7\phantom{0}}} = \frac{1}{\sqrt{119}} \approx 0.092$$

💡 ¡Observa! El Nodo 3 (grado 6) tiene el coeficiente más alto (0.092) porque tiene menos conexiones. El self-loop tiene el más bajo (0.059) porque $\hat{d}_0 = 17$ aparece dos veces. Esto es justo: los mensajes de nodos "menos ocupados" pesan más.

🔄 Paso 2 — Transformar cada vector con $\mathbf{W}$

La matriz $\mathbf{W}$ (de $34 \times 3$) transforma cada vector de 34 dimensiones a 3 dimensiones. Es la misma $\mathbf{W}$ para todos los nodos — la red aprende UNA transformación compartida.

$$\mathbf{W} \cdot \mathbf{h}_0 = \mathbf{W} \cdot [1,0,0,\ldots]^T = \text{columna 0 de } \mathbf{W} = [w_{00}, w_{01}, w_{02}]$$ $$\mathbf{W} \cdot \mathbf{h}_1 = \text{columna 1 de } \mathbf{W} = [w_{10}, w_{11}, w_{12}]$$

Como usamos la matriz identidad como entrada, $\mathbf{W} \cdot \mathbf{h}_j$ simplemente selecciona la columna $j$ de $\mathbf{W}$. ¡La multiplicación es trivial en nuestro caso!

➕ Paso 3 — Sumar todos los mensajes normalizados

Juntamos todo. Usamos los $c_{i,j}$ que calculamos en el Paso 1:

$$\mathbf{h}_0^{\prime} = \sigma\Big( \underbrace{c_{0,0}}_{\approx\,0.059} \cdot \mathbf{W}\mathbf{h}_0 + \underbrace{c_{0,1}}_{\approx\,0.077} \cdot \mathbf{W}\mathbf{h}_1 + \underbrace{c_{0,2}}_{\approx\,0.073} \cdot \mathbf{W}\mathbf{h}_2 + \underbrace{c_{0,3}}_{\approx\,0.092} \cdot \mathbf{W}\mathbf{h}_3 + \ldots \Big)$$

Recuerda: cada $c_{0,j} = \frac{1}{\sqrt{\hat{d}_0 \cdot \hat{d}_j}}$ — es el factor de la fórmula original.

El resultado es un vector de 3 números que "resume" quién es el Nodo 0 considerando su vecindario. Personas en el mismo grupo social terminarán con vectores parecidos → ¡fáciles de clasificar!

🎯 En resumen: GCNConv toma la "identidad" de cada persona (34 números), pregunta a sus amigos cómo son, promedia las respuestas dándole más peso a los amigos "menos ocupados", y produce un nuevo vector de 3 números → el embedding.

🚶 Los 3 Pasos de GCNConv (Framework "Message Passing")

Internamente, PyTorch Geometric implementa GCNConv usando el paradigma de Message Passing. Es una forma elegante de pensar en la fórmula anterior como 3 pasos secuenciales:

📨 Paso 1: MESSAGE — "Cada nodo prepara su carta"

Cada nodo $j$ toma su vector actual $\mathbf{h}_j$ y lo transforma usando la matriz de pesos $\mathbf{W}$:

$$\text{mensaje}_j = \mathbf{W} \cdot \mathbf{h}_j$$

Esto convierte un vector de 34 dimensiones en uno de 3 dimensiones. Es como comprimir la información de cada persona en un resumen muy compacto.

🥋 Karate Club: El Nodo 1 tiene $\mathbf{h}_1 = [0,1,0,0,\ldots]$ (34 nums). Después: $\mathbf{W} \cdot \mathbf{h}_1 = [0.42, -0.18, 0.73]$ (3 nums).
¡Ese es su "mensaje" para enviar a sus vecinos!

📬 Paso 2: AGGREGATE — "Cada nodo recoge las cartas de sus amigos"

El nodo $i$ recibe los mensajes de todos sus vecinos + el suyo propio y los suma. Pero antes de sumar, cada mensaje se escala con el factor de normalización:

$$\text{agregado}_i = \sum_{j \in \mathcal{N}(i) \cup \{i\}} \frac{1}{\sqrt{\hat{d}_i \cdot \hat{d}_j}} \cdot \text{mensaje}_j$$

¿Por qué normalizar? Piénsalo así:

❌ Sin normalizar

Mr. Hi (16 amigos) recibe 17 mensajes → vector ENORME.
Nodo 11 (1 amigo) recibe 2 mensajes → vector pequeño.
¡Los vectores no son comparables!

✅ Con normalizar

Mr. Hi: cada mensaje se divide por ~13 → vector moderado.
Nodo 11: cada mensaje se divide por ~4 → vector moderado.
¡Todos los vectores quedan en la misma escala!

🔄 Paso 3: UPDATE — "Actualizar mi representación"

El resultado de la suma normalizada se convierte en el nuevo vector del nodo. Opcionalmente se le suma un bias $\mathbf{b}$:

$$\mathbf{h}_i^{\prime} = \sigma\left(\text{agregado}_i + \mathbf{b}\right)$$

$\mathbf{h}_i^{\prime}$ ya no es solo la identidad del nodo. Ahora contiene información mezclada del nodo Y de todo su vecindario. Es un vector de 3 números que puede visualizarse en 3D (¡lo veremos pronto!).

📊 Ejemplo Visual: Nodo 0 recibe mensajes

El Nodo 0 tiene 16 vecinos (aquí mostramos solo 4). Recibe mensajes transformados por $\mathbf{W}$ y escalados por $c_{ij} = \frac{1}{\sqrt{\hat{d}_i \cdot \hat{d}_j}}$, más su propio mensaje (self-loop).
El resultado $\mathbf{h}_0'$ es un vector de 3 números que "resume" todo el vecindario de Mr. Hi.

📖 Definición Formal de GCNConv (paso a paso)

Cuando escribimos GCNConv(34, 3) en PyTorch Geometric, internamente se define:

hi(l+1) = σ( Σj∈N(i)∪{i}   cij · W(l) · hj(l) )

donde cij = 1 / √(d̂i · d̂j) y d̂i = grado(i) + 1

1
Parámetro in_channels=34: Cada nodo entra con un vector de 34 dimensiones (su identidad). Esto define las columnas de la matriz W.
2
Parámetro out_channels=3: La salida será un vector de 3 dimensiones (embedding). Esto define las filas de W, que es una matriz de 34×3 = 102 pesos aprendibles.
3
Self-loop automático (add_self_loops=True): Por defecto, GCNConv añade una conexión de cada nodo consigo mismo. Esto garantiza que al agregar mensajes de vecinos, el nodo también considere su propia información.
4
Normalización simétrica (normalize=True): Divide cada mensaje por 1/√(d̂i · d̂j). Esto evita que nodos muy conectados (como Mr. Hi con 16 conexiones) dominen la señal. Sin normalización, el embedding de Mr. Hi sería desproporcionadamente grande.
🔑 En resumen para el Club de Karate:

Cada persona (nodo) del club "pregunta" a sus amigos (vecinos) cómo son, recibe sus vectores transformados por W, los promedia ponderándolos según cuántas conexiones tiene cada uno, y el resultado es un nuevo vector de 3 números que "resume" quién es esa persona según su entorno social. Personas con amigos similares terminarán con embeddings similares → ¡fáciles de clasificar!

⚙️ Parámetros de Nuestra GCNConv(34, 3)
Matriz W:

Tamaño: 34 × 3 = 102 parámetros
Transforma de 34 dims → 3 dims

Bias (opcional):

Tamaño: 3 parámetros (si está activado)
Se suma al final de cada operación

Total en GCNConv: ~102-105 parámetros aprendibles

⚡ CAPA 3: ReLU — La Función de Activación

🤔 ¿Qué es ReLU?

Rectified Linear Unit

Es la función más simple del mundo:
ReLU(x) = max(0, x)

Si x es negativo → devuelve 0
Si x es positivo → devuelve x sin cambios

🎯 ¿Para qué sirve?

Sin ReLU, apilar capas no tendría sentido:
Linear + Linear = 1 Linear (matemáticamente)

ReLU "rompe" la linealidad, permitiendo que la red aprenda patrones más complejos.

Es como añadir "curvas" a un dibujo hecho solo con líneas rectas.

📈 La Gráfica de ReLU
Antes: [-1.5, 2.3, -0.8]
Vector con valores negativos
Después: [0, 2.3, 0]
Negativos convertidos a 0
🆚 ¿Por qué ReLU y no otra función?
Función Pros Contras
ReLU Rápida, simple, funciona muy bien "Neuronas muertas" si siempre <0
Sigmoid Salida entre 0 y 1 Gradientes muy pequeños
Tanh Salida entre -1 y 1 Más lenta que ReLU

🏷️ CAPA 4: Linear — El Clasificador Final

Esta es la capa más "tradicional". Toma los embeddings de 3 dimensiones y produce 4 scores (uno por clase).

🔢 La Operación

y = W × x + b

W: Matriz de pesos (3×4)
x: Vector de entrada (3 dims)
b: Bias/sesgo (4 valores)
y: Output scores (4 valores)

📊 Parámetros

Matriz W: 3 × 4 = 12 pesos
Vector b: 4 biases

Total: 16 parámetros
(Mucho menos que GCNConv)

📐 Visualización de la Transformación
💡 ¿Cómo interpreta la red estos scores?

Los scores NO son probabilidades todavía. Son "logits" — valores crudos.

• Score alto (ej: 2.3) → La red está "segura" de que es esa clase
• Score bajo (ej: -0.5) → La red está "insegura" de esa clase
• Score negativo → Evidencia en contra de esa clase

Durante el entrenamiento, usamos CrossEntropyLoss que convierte estos scores a probabilidades internamente.

🤔

🧐 Pregunta Clave: ¿Por qué la última capa es Linear y NO Softmax?

Esta es una de las dudas más comunes. Observa que nuestro forward termina así:

# Nuestro forward:
h = self.gcn(x, edge_index).relu()   # GCNConv + ReLU
z = self.out(h)                     # Linear(3→4) ← ¡Sin softmax!
return h, z                        # Retornamos logits CRUDOS
❌ Lo que parece lógico (INCORRECTO)
z = self.out(h)
probs = softmax(z)   # ¡NO hagas esto!
loss = CrossEntropyLoss(probs, y)

⚠️ Doble softmax → CrossEntropyLoss YA aplica softmax internamente. Si tú también lo aplicas, haces softmax(softmax(z)), lo que produce gradientes muy pequeños y el modelo no aprende bien.

✅ Lo correcto (LO QUE HACEMOS)
z = self.out(h)      # Logits crudos
loss = CrossEntropyLoss(z, y)  # ✅

✅ CrossEntropyLoss = LogSoftmax + NLLLoss. Ya incluye softmax internamente, así que solo necesita los logits crudos.

📋 Las 3 Razones Técnicas:
1
Estabilidad numérica

Softmax calcula ex, que puede explotar a valores enormes (overflow) o colapsar a cero (underflow). PyTorch usa un truco llamado LogSumExp internamente en CrossEntropyLoss que evita estos problemas. Si aplicas softmax tú mismo, pierdes esta protección.

2
Gradientes más sanos

Con logits crudos, el gradiente de CrossEntropyLoss es simplemente prob_predicha - 1 para la clase correcta. Si haces doble softmax, los gradientes se "aplastan" y el entrenamiento se vuelve extremadamente lento o se estanca.

3
Para predecir, NO necesitas softmax

Al momento de predecir, usamos argmax (el índice del valor más alto). Y argmax de los logits = argmax de softmax(logits), porque softmax preserva el orden. ¡Así que es innecesario calcularlo!

🔄 ¿Cuándo se aplica Softmax entonces?
💡
Regla de oro en PyTorch: Si usas nn.CrossEntropyLoss, tu modelo debe devolver logits crudos (sin softmax). Si en cambio usas nn.NLLLoss, entonces sí debes aplicar F.log_softmax() tú mismo. En nuestro caso, usamos CrossEntropyLoss → la capa final es Linear sin activación.

🎯 CAPA 5: La Predicción Final

El último paso: convertir los 4 scores en una predicción concreta de clase.

1️⃣ Durante Entrenamiento

CrossEntropyLoss aplica internamente:

1. Softmax: Convierte scores a probabilidades (suman 1)
2. Log: Toma el logaritmo
3. NLL: Compara con la clase real

Loss = -log(prob_clase_correcta)

2️⃣ Durante Predicción

Usamos argmax — el índice del score más alto:

z.argmax(dim=1)

• Si scores = [2.3, 0.8, -0.5, 1.2]
• argmax = 0 (porque 2.3 es el mayor)
• Predicción: Clase 0

📊 De Scores a Probabilidades (Softmax)

Softmax = escore / Σescores → Todas las probabilidades suman 100%

# Función de pérdida y optimizador
criterion = torch.nn.CrossEntropyLoss()
optimizer = torch.optim.Adam(model.parameters(), lr=0.02)

# Función de precisión
def accuracy(pred_y, y):
    return (pred_y == y).sum() / len(y)

# Loop de entrenamiento
for epoch in range(201):
    optimizer.zero_grad()           # 1. Limpiar gradientes
    h, z = model(data.x, data.edge_index)  # 2. Forward pass
    loss = criterion(z, data.y)     # 3. Calcular pérdida
    acc = accuracy(z.argmax(dim=1), data.y)  # 4. Calcular precisión
    loss.backward()                 # 5. Calcular gradientes
    optimizer.step()                # 6. Actualizar pesos
    
    if epoch % 10 == 0:
        print(f'Epoch {epoch:>3} | Loss: {loss:.2f} | Acc: {acc*100:.2f}%')

🔬 ¿Qué pasa en cada línea del loop? (Para los curiosos)

No te preocupes si no entiendes todo esto al 100%. Lo importante es la intuición. Pero si quieres profundizar, aquí está el detalle de cada paso:

1

optimizer.zero_grad() — Limpiando el Pizarrón

💡 Analogía del Pizarrón

Imagina que estás resolviendo problemas de matemáticas en un pizarrón. Antes de cada nuevo problema, borras el pizarrón para no confundirte con los cálculos anteriores. ¡Eso es exactamente lo que hace zero_grad!

2

model(data.x, data.edge_index) — El Pase Hacia Adelante

3

criterion(z, data.y) — Midiendo el Error con CrossEntropyLoss

Loss = -Σᵢ yᵢ · log(softmax(zᵢ))
y = clase verdadera (one-hot) | z = predicciones del modelo | softmax convierte a probabilidades

💡 ¿Por qué CrossEntropy?

Cross Entropy es perfecta para clasificación porque:
• Penaliza fuertemente cuando el modelo está muy seguro de algo incorrecto
• Penaliza suavemente cuando el modelo está inseguro pero correcto
• Incluye Softmax internamente, convirtiendo scores a probabilidades

4

loss.backward() — Backpropagation: El Camino de Regreso

💡 Analogía de la Investigación

Imagina que un equipo de cocina hizo un platillo horrible. El jefe de cocina (loss.backward) investiga: "¿Quién puso demasiada sal?"

Revisa cada paso hacia atrás: ¿Fue el emplatado? ¿La cocción? ¿El sazonado? ¿Los ingredientes?
Así se calcula la "culpabilidad" (gradiente) de cada parámetro.

5

optimizer.step() — Actualizando los Pesos con Adam

W_nuevo = W_actual - lr × gradiente
lr = learning rate (0.02) | gradiente = dirección de máximo aumento del error

🎓 ¿Por qué Adam y no otro optimizer?

📉 SGD Simple

W = W - lr × grad

Puede ser lento y oscilante. Usa el mismo learning rate para todo.

🚀 Adam (Adaptive Moment)

Combina Momentum + RMSprop. Se adapta a cada parámetro.

Es el "go-to" optimizer para la mayoría de casos.
